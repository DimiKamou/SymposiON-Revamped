III Findings — the panel & its content

high

Ενότητα 26 ships the text of Ενότητα 27

content
games/latin-texts/units/unit26.js

unit26.js (labelled “Lectio XXVI”) contains the same Accius/Pacuvius passage as unit27.js — both tiles are even titled “Το πνεύμα ωριμάζει όπως… οι καρποί.” The two files then disagree on identical tokens (e.g. quidem: “βεβαιωτικό μόριο” vs “τροπικό επίρρημα”; lemma of multo). A student opening Ενότητα 26 gets Lectio 27, and the genuine Lectio XXVI material is absent.

Fix: replace unit26.js with the real Lectio XXVI text (or drop the tile), and reconcile the divergent per-token analyses.
med

The Εισαγωγή tile is dead — opener never registered

functional · in-scope
js/manifest/latin-texts.js:14

The manifest loop registers SYN_GAMES['openLatinText'+n] for every unit but never openLatinIntro. Both the dispatcher (screens.js:70) and synLaunch() bail when SYN_GAMES[fn] is missing — so clicking the “Εισαγωγή / Introduction” tile does nothing (console: “no manifest entry”). Verified against the actual launch path.

Fix (1 line): add G['openLatinIntro'] = entry; to the manifest loop.
med

A single missing array field blanks the whole panel

robustness
games/latin-texts/app/panel.jsx:383–413

renderVals() reads seven parts as (ready?u.nouns:[]).map(…) — the ternary only guards a null u. If an author ships a unit missing verbs/nouns/alignment (easy when copying a template), .map throws inside render() with no error boundary → white screen. Current data is clean, so this is latent.

Fix: coerce each with Array.isArray(...)?...:[] (mirroring the existing transforms/etymology guard) and default nested arrays.
low

Admin mode restores from a plain localStorage flag — PIN offers no protection

client-security · known
games/latin-texts/app/panel.jsx:124 · 200

Setting localStorage.latinAdmin='1' unlocks teacher/edit mode on reload with no PIN prompt (confirmed live). Impact is contained: edits persist only to the local browser and export a .js file — nothing server-side. The PR already flags the PIN as a placeholder pending the real teacher account.

Fix: acceptable as an interim, but swap the static flag for the authenticated teacher session before launch (as planned).
low

Pinned word-popup doesn’t follow the text on scroll

UX
games/latin-texts/app/panel.jsx:285 · 443

A click-pinned analysis card is position:fixed at captured viewport coordinates; scrolling the passage leaves it stranded over unrelated text until dismissed.

Fix: reposition against the stored element on scroll, or auto-unpin when the container scrolls.

IV Security — where the real risk lives

The Latin PR adds no backend, so the panel’s own surface is clean. The security-relevant findings are all pre-existing in the app that ships around it — surfaced here because you asked for a security pass, and worth fixing regardless of #72.

high

Any signed-in user can self-grant “Pro” access — payment bypass

pre-existing · Firestore rules
firestore.rules:75

The users/{uid} self-update rule pins only role (request.resource.data.role == resource.data.role) and leaves plan, expiresAt, classes unconstrained — the exact fields the Stripe webhook writes to mark someone paid. A free user can setDoc(merge) their own doc with plan:'pro' + all classes + a far-future expiry and the client treats them as a paid subscriber. Mitigant: paid content docs are world-readable anyway, so this forges client-trusted entitlement/billing UI rather than exposing otherwise-protected data.

Fix: pin the entitlement fields the way role is pinned (or make them Admin-SDK-only), and gate Pro off a custom claim clients can’t forge.
med

config/* (incl. pricing) is directly writable by staff roles

pre-existing · Firestore rules
firestore.rules:179

allow write: if can('subs') || can('content') lets any holder of those custom-claim roles write config/pricing straight from the client, bypassing the adminSaveConfig validation (no €0 paid plans, price caps, coupon bounds). A content editor could set every plan to €0.01.

Fix: set allow write: if false and route all config writes through the validated Cloud Function (as siteCatalog/gameContent already do).
med

gradeAnswer is an open, unauthenticated proxy to the Anthropic API

pre-existing · Cloud Function
functions/index.js:1016

onRequest with Access-Control-Allow-Origin:*, no context.auth, no App Check, no rate limit — it forwards any POST to Claude using the server’s ANTHROPIC_KEY. Anyone who finds the URL can burn your token quota / bill. (The comment assumes same-origin in prod, but CORS * permits cross-origin.)

Fix: require Firebase Auth / App Check, drop CORS to the site origin, and add per-user rate limiting.
